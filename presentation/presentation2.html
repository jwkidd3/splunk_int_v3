Module 7

Creating & Managing Fields

Understanding Field Extractions

What are Field Extractions?

  • Extract fields from raw event text
  • Create structured data from unstructured logs
  • Applied at search time, not index time

Example: Extract IP address and port from Linux secure logs

Two Extraction Methods

Regex Extraction

For semi-structured text

  • Linux logs
  • Syslog
  • Custom app logs

Delimiter Extraction

For delimited data

  • CSV files
  • TSV files
  • Pipe-delimited

Tool: Field Extractor (FX) generates patterns automatically

Field Extractor - Regex Method

Sample Event:

Failed password for admin from 192.168.1.100 port 52394 ssh2

Steps

  1. Event Actions → Extract Fields
  2. Select method: Regular Expression
  3. Highlight IP → Field name: src_ip
  4. Highlight port → Field name: port
  5. Validate → Save

⏱️ Wait ~1 minute for knowledge bundle replication

Module 8

Field Aliases & Calculated Fields

Understanding Field Aliases

The Problem: Different field names across sourcetypes

  • cisco_wsa_squid → cs_username
  • linux_secure → user
  • cisco_firewall → Username

Solution: Field aliases create consistent naming

Settings → Fields → Field aliases

Creating Field Aliases

# Configuration
Name: cisco_wsa_username_alias
Apply to: sourcetype=cisco_wsa_squid
Field alias: user = cs_username

# Usage
(index=network sourcetype=cisco_wsa_squid) OR
(index=security sourcetype=linux_secure)
| stats count by user, sourcetype
# Now 'user' works across both sourcetypes!

Calculated Fields

Automatically compute values at search time

Settings → Fields → Calculated fields

Example: Convert bytes to megabytes

Name: cisco_wsa_megabytes
Sourcetype: cisco_wsa_squid
Eval: round(sc_bytes/(1024*1024), 2)
Save as field: sc_megabytes

# Result: sc_megabytes appears automatically in all searches!

Field Aliases vs. Calculated Fields

Field Aliases

  • Normalize field names
  • No computation
  • CIM compliance
  • Example: user = cs_username

Calculated Fields

  • Compute new values
  • Requires eval expression
  • Automatic calculations
  • Example: megabytes = bytes/1024/1024

Module 9

Tags & Event Types

Understanding Tags

What are Tags? Labels for field values

The Problem: Multiple admin user variants

  • admin, administrator, sysadmin, itmadmin, sapadmin

Solution: Tag all as privileged_user

Creating Tags

# Method 1: From Search Results
1. Search: index=security user=admin
2. Fields sidebar → Click user → Click ⋯ next to "admin"
3. Edit Tags → Enter: privileged_user
4. Repeat for other admin variants

# Method 2: Settings → Tags
- Add multiple field/value pairs at once

Using Tags

# Instead of this:
index=security (user=admin OR user=administrator OR
                user=sysadmin OR user=itmadmin)

# Use this:
index=security tag=privileged_user
| stats count by user

# Failed logins by privileged users
index=security tag=privileged_user "failed password"
| stats count by user, src_ip
| sort -count

Understanding Event Types

What are Event Types?

  • Save search strings as reusable objects
  • Define categories of events
  • Can include multiple criteria and sourcetypes
  • Assigned priorities for overlapping matches

Event Type vs Tag: Event Type = complex search criteria | Tag = field value labels

Creating Event Types

# Example: Web Error Event Type
Name: web_error
Search string: (sourcetype=access_combined OR
                sourcetype=cisco_wsa_squid) status>500
Priority: 1 (Highest)
Tags: error, web, server_error

# Usage
index=web eventtype=web_error
| stats count by status, sourcetype

Module 10

Creating & Using Macros

Understanding Search Macros

What are Search Macros?

  • Reusable search components
  • Encapsulate complex logic
  • Accept arguments for flexibility
  • Improve consistency and maintainability

Invocation: Use backticks `macro_name`

Creating Basic Macros

# Settings → Advanced search → Search macros
Name: Europe_sales
Definition: sourcetype=vendor_sales
            VendorCountry IN (Germany, France, Italy)

# Usage
`Europe_sales`
| stats sum(price) as revenue by product_name
| sort -revenue

Macros with Arguments

# Currency Conversion Macro
Name: convert_sales(3)  ← (3) = 3 arguments
Arguments: currency, symbol, rate
Definition:
| stats sum(price) as USD by product_name
| eval $currency$ = "$symbol$" +
       tostring(round(USD*$rate$, 2), "commas")

# Usage
`Europe_sales`
`convert_sales(euro,€,0.79)`
| table product_name USD euro

Macro Validation

# Add validation to convert_sales(3)
Validation Expression: isnum($rate$)
Validation Error: Exchange rate must be numeric

# Test
`convert_sales(euro,€,0.79)`  ✓ Works
`convert_sales(euro,€,invalid)`  ✗ Error shown

Validation Functions: isnum(), isnotnull(), match()

Module 11

Workflow Actions

Understanding Workflow Actions

GET Actions

Open external URLs

  • WHOIS lookups
  • Google searches
  • Threat intel portals

POST Actions

Submit data to systems

  • Create tickets
  • Block IPs
  • Send alerts

Search Actions

Launch new Splunk searches

  • Investigate related events | Drill into user activity | Correlation searches

Creating GET Workflow Actions

# WHOIS IP Lookup
Settings → Fields → Workflow actions

Name: get_whois_info
Label: Get WHOIS info for IP: $src_ip$
Apply to field: src_ip
Action type: link
Link method: get
URI: http://who.is/whois-ip/ip-address/$src_ip$
Open in: New window

# Usage: Click event → Click src_ip → Select workflow action

Creating Search Workflow Actions

# Failed Login Investigation
Name: search_failed_login_by_ip
Label: Search failed logins by IP: $src_ip$
Apply to field: src_ip
Action type: search
Search string:
  index=security sourcetype=linux_secure
  "failed password" src_ip=$src_ip$
  | stats count by user, src_ip | sort -count
Earliest: -7d
Latest: now

# Result: Click IP → New search shows all failed logins

Module 12

Creating Data Models

Understanding Data Models

What are Data Models?

  • Hierarchical representation of datasets
  • Parent-child relationships
  • Enable Pivot interface for non-SPL users
  • Can be accelerated for performance

Components: Root Event Dataset | Child Datasets | Fields (auto-extracted, eval, lookup)

Buttercup Games Site Activity

Data Model Structure:
Buttercup Games Site Activity (Root)
└── Web requests (Root Event)
    ├── Successful requests (status<400)
    │   └── purchases (action=purchase productId=*)
    └── Failed requests (status>399)
        └── removed (action=remove productId=*)

Purpose: Enable marketing team to analyze website
         activity without knowing SPL

Creating Root Event Dataset

# Settings → Data models → New Data Model
Title: Buttercup Games Site Activity
ID: buttercup_games_site_activity

# Add Dataset → Root Event
Name: Web requests
ID: web_requests
Constraints: index=web sourcetype=access_combined

# Add Auto-Extracted Fields
action, bytes, categoryId, clientip, productId,
product_name, req_time, status

Adding Eval Expression Fields

# Day of Week Field
Field: day
Eval: strftime(_time, "%m-%d %A")
Type: String

# Megabytes Field
Field: megabytes
Eval: round(bytes/(1024*1024), 2)
Type: Number

# Response Category
Field: response_category
Eval: case(
  req_time<0.1, "Fast",
  req_time>=0.1 AND req_time<0.5, "Normal",
  req_time>=0.5, "Slow"
)
Type: String

Using the Pivot Interface

Configure

  • Split Rows
  • Split Columns
  • Values (metrics)
  • Filters

Output

  • Tables
  • Charts
  • Statistics
  • Trends

Point-and-click data analysis - no SPL required!

Data Model Acceleration

What is Acceleration?

Pre-computed summary tables for instant results

  • 10-100x faster pivot reports
  • Enable tstats command
  • Trade storage for speed
| tstats count from datamodel=buttercup_games_site_activity
  by Web_requests.method, Web_requests.status
