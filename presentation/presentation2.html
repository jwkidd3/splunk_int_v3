Creating and Managing Fields

Field Extraction Methods

Regular Expression

  • Unstructured data
  • Pattern-based
  • Example: Extract IP from logs
from 192.168.1.100 port 52394

Delimiter

  • Structured data (CSV)
  • Simpler to configure
  • Example: Game telemetry
time,src,version,user,action

Field Extractor (FX)

UI-driven field extraction tool

Steps

  1. Event Actions â†’ Extract Fields
  2. Choose method (regex or delimiter)
  3. Highlight field values or select delimiter
  4. Name fields
  5. Validate extraction
  6. Save (wait ~1 min for replication)

Regex Extraction Example

Extract src_ip and port from linux_secure logs

index=security sourcetype=linux_secure

Sample event: "from 192.168.1.100 port 52394"

Extraction:

  • Highlight "192.168.1.100" â†’ name: src
  • Highlight "52394" â†’ name: port

Splunk generates: from\s+(?<src>[^\s]+)\s+port\s+(?<port>\d+)

Delimiter Extraction Example

Extract fields from SimCubeBeta game logs

2024-01-15T10:30:45,192.168.50.25,v1.2.3,misc,user123,DragonSlayer,attack,warrior

Select delimiter: Comma

Name fields: time, src, version, misc, user, CharacterName, action, role

Much simpler than regex for structured data!

Field Aliases & Calculated Fields

Field Aliases

Normalize field names across sourcetypes

Problem: Different field names for same data

  • cisco_wsa_squid: cs_username
  • linux_secure: user

Solution: Create alias user = cs_username

Creating Field Aliases

Settings â†’ Fields â†’ Field aliases

Name: cisco_wsa_username_alias
Apply to: sourcetype=cisco_wsa_squid
Field alias: user = cs_username

Result: Both user and cs_username work in searches

No data duplication - just alternate names

Calculated Fields

Automatically compute values using eval expressions

Name: cisco_wsa_megabytes
Apply to: sourcetype=cisco_wsa_squid
Eval expression: round(sc_bytes/(1024*1024), 2)
Save as field: sc_megabytes

Benefit: Field always available, no manual eval needed

Applied at search time automatically

When to Use Each

Use Case Use Example
Different names, same data Field Alias cs_username â†’ user
Unit conversion Calculated Field bytes â†’ megabytes
Data classification Calculated Field req_time â†’ response_category
CIM compliance Both Normalize field names & values

Tags & Event Types

Tags

Create meaningful labels for field-value pairs

Problem: Multiple admin user variations

  • admin
  • administrator
  • sysadmin
  • itmadmin

Solution: Tag all as privileged_user

Creating Tags

Settings â†’ Tags â†’ New Tag

Tag Name: privileged_user
Field/Value pairs:
  - user = admin
  - user = administrator
  - user = sysadmin
  - user = itmadmin
  - user = sapadmin

Search: tag=privileged_user

Simpler than: user=admin OR user=administrator OR...

Event Types

Save search strings as reusable objects

Name: web_error
Search string: (sourcetype=access_combined OR sourcetype=cisco_wsa_squid) status>500
Priority: 1 (Highest)
Tags: error, web, server_error

Search: eventtype=web_error

Or use tags: tag=error

Tags vs Event Types

Tags Event Types
Purpose Label field values Save search patterns
Scope Single field-value Multiple conditions
Example user=admin â†’ privileged_user status>500 â†’ web_error
Search tag=tagname eventtype=typename

Creating and Using Macros

What Are Macros?

Reusable search components

  • Encapsulate search logic
  • Can accept arguments
  • Invoked with backticks: `macro_name`
  • Improve consistency and maintainability

Basic Macro

Settings â†’ Advanced Search â†’ Search macros

Name: Europe_sales
Definition: sourcetype=vendor_sales VendorCountry IN (Germany, France, Italy)

Usage:

`Europe_sales`
| stats sum(price) as USD by product_name
| sort -USD

Macros with Arguments

Accept parameters for flexibility

Name: convert_sales(3)
Arguments: currency, symbol, rate
Definition: | stats sum(price) as USD by product_name | eval $currency$="$symbol$"+tostring(round(USD*$rate$,2),"commas")

Usage:

sourcetype=vendor_sales VendorCountry IN (Germany, France, Italy)
`convert_sales(euro,â‚¬,0.79)`
| table product_name USD euro

Macro Validation

Ensure arguments meet requirements

Validation Expression: isnum($rate$)
Validation Error Message: Exchange rate must be a numeric value

Common validation functions:

  • isnum($arg$)
  • isnotnull($arg$)
  • match($arg$, "pattern")

Nested Macros

Macros can call other macros

Europe_sales: sourcetype=vendor_sales VendorCountry IN (Germany, France, Italy)
Asia_sales: sourcetype=vendor_sales VendorCountry IN (Japan, China, South Korea)
Americas_sales: sourcetype=vendor_sales VendorCountry IN (United States, Canada, Mexico)

regional_comparison: ( `Europe_sales` ) OR ( `Asia_sales` ) OR ( `Americas_sales` )

Usage:

`regional_comparison`
| stats sum(price) as total_sales by VendorCountry

Workflow Actions

Workflow Action Types

GET

Open external URLs

Example: WHOIS lookup

http://who.is/whois-ip/ip-address/$src_ip$

POST

Submit data to APIs

Example: Create ticket

priority=high&ip=$src_ip$

Search

Launch new Splunk search

index=security src_ip=$src_ip$ | stats count by user

Creating GET Workflow Action

Settings â†’ Fields â†’ Workflow actions

Name: get_whois_info
Label: Get WHOIS info for IP: $src_ip$
Apply to: src_ip field
Action type: link
Link method: get
URI: http://who.is/whois-ip/ip-address/$src_ip$
Open link in: New window

Click on src_ip in any event â†’ "Get WHOIS info" appears

Creating Search Workflow Action

Enable contextual investigations

Name: search_failed_login_by_ip
Label: Search failed logins by IP: $src_ip$
Apply to: src_ip field
Action type: search
Search string: index=security sourcetype=linux_secure "failed password" src_ip=$src_ip$ | stats count by user, src_ip | sort -count
Time range: Last 7 days

One-click investigation workflows

Workflow Action Benefits

  • Faster investigations: Pre-built contextual searches
  • External integration: Link to ticketing, WHOIS, threat intel
  • Consistency: Standardized investigation workflows
  • User-friendly: Click instead of typing searches
  • Multi-step workflows: Chain multiple actions

Creating Data Models

What Are Data Models?

Hierarchical representation of data

  • Organize data into parent-child relationships
  • Enable Pivot interface (no SPL required)
  • Can be accelerated for performance
  • Standardize field definitions

Data Model Components

Datasets

  • Root Event: Base search
  • Child: Subset with constraints
  • Grandchild: Further refinement

Fields

  • Auto-extracted: From events
  • Eval: Calculated fields
  • Lookup: Enrichment
